# Act While Thinking: Accelerating LLM Agents via Pattern-Aware Speculative Tool Execution

**Authors**: Yifan Sui (Shanghai Jiao Tong University), Han Zhao (Shanghai Jiao Tong University), Rui Ma (Microsoft Research), Zhiyuan He (Microsoft Research), Hao Wang (Stevens Institute of Technology), Jianxun Li (Shanghai Jiao Tong University), Yuqing Yang (Microsoft Research)

**Submitted**: arXiv, 19 Mar 2026

**Subjects**: Distributed, Parallel, and Cluster Computing (cs.DC); Artificial Intelligence (cs.AI)

## Abstract

LLM-powered agents are emerging as a dominant paradigm for autonomous task solving. Unlike standard inference workloads, agents operate in a strictly serial "LLM-tool" loop, where the LLM must wait for external tool execution at every step. This execution model introduces severe latency bottlenecks.

To address this problem, we propose **PASTE** (Pattern-Aware Speculative Tool Execution), a method designed to hide tool latency through speculation. PASTE exploits two key observations: (1) agent requests exhibit stable application-level control flows (recurring tool-call sequences), and (2) predictable data dependencies (parameter passing between tools).

**Results**: PASTE reduces average task completion time by **48.5%** and improves tool execution throughput by **1.8x** compared to state-of-the-art baselines.

## 1. Introduction

### The Era of LLM Agents

LLM agents reframe the model as the cognitive core of a broader control loop. An agent is defined not just by its underlying model, but by its ability to orchestrate multi-step tasks through the use of external tools (Python interpreters, web retrieval APIs, vector databases) and long-term memory.

Unlike traditional inference workloads that are relatively short and independent, agent workloads are **long-running, stateful, and highly sequential**. A single user request (e.g., "Analyze this dataset and plot the trends") triggers many intermediate reasoning steps and tool executions that may span minutes.

### The Agent Execution Model

LLM agents follow an *Iterative LLM–Tool Loop*:
1. LLM is conditioned on current context and execution history
2. LLM either emits final answer and terminates, OR decides to invoke a tool with concrete parameters
3. If tool is invoked: system executes call, appends tool output to session state, resumes LLM inference on updated context
4. Loop repeats until LLM produces final result

### Problem: Tool Execution is the Bottleneck

Experimental results on three mainstream agent benchmarks (Deep Research Bench, SWE-Bench, ScholarQA) show:
- **Tool execution accounts for 35%-61% of total request time**
- Coding tasks: 60% tool latency
- Deep research tasks: 50% tool latency
- Scientific tasks: 36% tool latency

The LLM generation and tool execution are **strictly serialized** because of inherent data dependencies. This makes tool execution the dominant latency bottleneck.

## 2. Key Observations and Insights

### Insight 1: Predictable Control Flow Patterns

Tool invocations follow distinct, domain-specific state transition probabilities. Two major patterns:

**In Coding (Bug-fix assistance):**
- **Edit-Verify Pattern**: 55% of successful file_editor tool calls are immediately followed by a terminal tool call (pytest or python execution)
- **Locate-Examine Pattern**: 38% of grep tool calls are immediately followed by file_editor tool call

**In Deep Research:**
- **Search-Visit Pattern**: 51% of search tool calls are immediately followed by web_fetch to visit top 3-4 URLs
- **Pre-fetch Pattern**: 20% of web_fetch calls trigger pre_fetching of embedded URLs

### Insight 2: Implicit Data Flow and Parameter Derivation

Not all tool parameters are "hallucinated" by the LLM from scratch—they could be derived from the output of previous tools.

**Producer-Consumer Pattern:**
- In web tasks: visit tool requires a URL; in 95% of cases, this URL is a strict substring of the JSON output from the preceding search call
- In coding tasks: file_editor requires a filename distilled directly from output of a prior grep call

**Repetitive-Debug Pattern:**
- After completing a fix, the agent immediately executes the program to verify the result (edit-verify dependency cycle)

**Implication**: Instead of generating arguments token-by-token (which is slow), PASTE can identify "candidate arguments" (URLs, filenames) from execution history and speculatively execute.

### Insight 3: Latent Parallelism in Serial Agent Execution

While standard agent frameworks (ReAct) force serial execution, significant parallelism opportunities exist:

- **Broad Search Pattern**: Once LLM determines keywords, agents often issue multiple search queries across Arxiv, PubMed, Google Scholar—could be parallelized
- **Batch Fetch Pattern**: Upon receiving search results, agents need to sequentially fetch landing pages, PDFs, code repositories—could be parallelized

## 3. System Design: PASTE

### Architecture Overview

PASTE consists of two core components:
1. **Pattern Abstraction** that decouples control flow from data flow
2. **Scheduler** that decouples speculative tool execution from authoritative tool paths

### 3.1 Pattern Tuple Abstraction

PASTE introduces the **Pattern Tuple (C, T, f, p)**:

- **C (Context)**: Event signatures (sequence of tool types) that identify stable control flows
- **T (Tool prediction)**: The predicted next tool to invoke
- **f (Function)**: A value mapping function that encodes logical relationships between data (parameter derivation)
- **p (Probability)**: Confidence score for the prediction

Example patterns:

| Pattern | Context (C) | Tool (T) | Function (f) | Prob (p) |
|---------|-------------|----------|--------------|----------|
| P1 | [(Search, success)] | Web_fetch | arg0 = SearchRes["list"][0]["url"] | 0.9 |
| P2 | [(Search, success), (Web_fetch, fail)] | Web_fetch | arg0 = SearchRes["list"][1]["url"] | 0.8 |

The abstraction achieves robustness by **separating execution structure from execution content**. While agents solve similar problems using different natural language phrasing, the tuple defines context strictly over event signatures.

### 3.2 Pattern Mining and Validation

Two-phase mining process:

**Phase I: Structural Mining (The "Where")**
- Identifies recurring tool-call sequences from historical agent execution traces
- Uses n-gram analysis to find frequent patterns

**Phase II: Symbolic Dependency Inference (The "How")**
- Infers parameter derivation functions (how output of tool N becomes input to tool N+1)
- Uses symbolic execution to capture data flow relationships

### 3.3 Online Pattern Prediction

When a pattern matches the current execution context:
1. **Context Matching**: Matches current tool sequence against known patterns
2. **Function Evaluation**: Applies the value mapping function to derive tool parameters
3. **Probabilistic Hint Generation**: Outputs predicted tool + parameters with confidence

### 3.4 Opportunistic Speculative Scheduling

PASTE employs **scheduling with opportunistic speculation**:

**Two job types:**
- **Authoritative invocations**: Generated by the agent, correctness-critical, given strict priority and preemption capability
- **Speculative invocations**: Generated by PASTE, best-effort, confined to a dynamic slack budget

**Key principles:**
- Speculative jobs run *only* on slack resources (transient idle compute/memory/IO capacity)
- Throttled by a dynamic slack budget—they harvest otherwise-wasted cycles without competing with normal LLM inference
- **Instantaneously preempted** when resource contention arises—mispredictions never slow down authoritative progress

**Promotion Protocol:**
When an authoritative request arrives and matches a running speculative job, that job is immediately promoted from low priority to high priority, and its result is committed directly to agent history.

## 4. Evaluation Results

### Experimental Setup
- Benchmarks: Deep Research Bench, SWE-Bench, ScholarQA
- Agent products: Gemini-2.5, GPT-5.2, Qwen-DeepResearch-30B
- Hardware: 8x NVIDIA A100-80G GPUs

### Key Results

| Metric | Improvement |
|--------|------------|
| Average task completion time | **-48.5%** |
| Tool execution throughput | **1.8x** |

### Time Breakdown Analysis
- Tool execution accounts for 35%-61% of total request time across benchmarks
- PASTE significantly reduces effective tool latency through speculation
- LLM generation and tool execution overlap achieved through speculative parallelization

### Pattern Prediction Accuracy
- High accuracy on Edit-Verify patterns (55% deterministic chains)
- Strong performance on Search-Visit patterns (51% deterministic)
- Value mapping functions achieve >90% accuracy in parameter derivation

## 6. Conclusion

PASTE addresses the latency bottleneck in LLM agent serving by exploiting predictable control flow patterns and data dependencies. The key innovations:

1. **Pattern Tuple abstraction** decouples control flow from data flow, enabling robust pattern matching across diverse agent sessions
2. **Opportunistic speculative scheduling** harvests slack resources without interfering with authoritative agent progress
3. **Symbolic parameter derivation** predicts tool arguments without expensive token-by-token generation

**Results**: 48.5% reduction in task completion time, 1.8x tool throughput improvement.
