## Main Idea

OpenViking is an open-source context database for AI agents that replaces fragmented vector storage with a file system paradigm—unifying memory, resources, and skills under a `viking://` virtual filesystem URI scheme with three-tier context loading (L0/L1/L2), directory recursive retrieval, visualized retrieval trajectories, and automatic session-based self-evolution.

## Key Details

**Core Innovation — Filesystem Paradigm**:
- All context (memories, resources, skills) mapped to virtual directories under `viking://` protocol
- Agents manipulate context via standard operations: `ls`, `find`, `tree`, `grep`
- URI examples: `viking://resources/`, `viking://user/memories/preferences/`, `viking://agent/skills/`

**Three-Tier Context Loading**:
- **L0 (Abstract)**: ~100 token summary for quick relevance check
- **L1 (Overview)**: ~2k tokens with core information for planning
- **L2 (Details)**: Full original data; loaded on demand

**Directory Recursive Retrieval**:
1. Intent analysis → multiple retrieval conditions
2. Initial positioning via vector retrieval
3. Refined exploration within high-score directory
4. Recursive drill-down into subdirectories
5. Result aggregation

**Visualized Retrieval Trajectory**: Every retrieval's directory browsing and file positioning is preserved; users observe exactly how context was located.

**Automatic Session Management**: After each session, memory extraction automatically updates User (preferences) and Agent (operational tips, tool experience) directories—agent gets smarter with use.

**VikingBot**: AI agent framework built on OpenViking (`pip install "openviking[bot]"`)

**VLM Providers**: volcengine (Doubao), openai, litellm (unified: Anthropic, DeepSeek, Gemini, Qwen, vLLM, Ollama, etc.)

**Embedding Providers**: volcengine, openai, jina, voyage, minimax, vikingdb, gemini

**OpenClaw Plugin Results** (LoCoMo benchmark):
| Group | Task Completion | Input Tokens |
|-------|----------------|--------------|
| OpenClaw (memory-core) | 35.65% | 24,611,530 |
| OpenClaw + LanceDB | 44.55% | 51,574,530 |
| OpenViking Plugin (–memory-core) | **52.08%** | 4,264,396 |
| OpenViking Plugin (+memory-core) | 51.23% | 2,099,622 |

- With native memory: 43% improvement over original OpenClaw, 91% token cost reduction
- Without native memory: 49% improvement over original OpenClaw, 83% token cost reduction

**OpenCode Memory Plugin**: Also available

**Installation**: `pip install openviking`; Rust CLI via `cargo install --git ... ov_cli`

**Deployment**: Standalone HTTP service; recommended on Volcengine ECS with veLinux

## Method / Approach

OpenViking departs from flat vector storage RAG by implementing a hierarchical virtual filesystem. When context is written, it's automatically processed into three tiers (L0/L1/L2) with summaries generated by VLM. On retrieval, the system first locates high-relevance directories via vector search, then recursively explores subdirectories, preserving full traversal history for observability.

## Results / Evidence

- LoCoMo benchmark: 52.08% task completion with OpenViking + OpenClaw (vs 35.65% baseline OpenClaw)
- Token cost reduction: 83–96% depending on configuration
- 49% improvement over baseline when OpenClaw's native memory is disabled

## Limitations

- AGPLv3 license (main project) may be restrictive for commercial use
- Requires Go 1.22+ and C++ compiler for building AGFS components—more complex setup than pure Python solutions
- Self-evolution loop quality depends on VLM summarization quality—garbage in, garbage out
